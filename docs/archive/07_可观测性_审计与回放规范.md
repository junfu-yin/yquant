# 07 · 可观测性、审计与回放规范

> yquant 阶段二文档 · 编号 07 · v1.0 · 2026-07-05
> 依赖：03 §5/§7；被 06（T11/T13）、08（Gate 证据）、09（评估数据来源）依赖。
> 立场（ADR-13）：**无凭证，不决策。** "能还原现场"不是靠事后打捞日志的运气，而是架构性质：任何进入账本的决策，写入那一刻就必须携带足以逐字节重演它的全部凭证，否则写入被拒绝。

---

## 1. 三层观测体系

1. **决策事件账本（本篇主角）**：append-only 的 `decision_events` 表，记录一切有业务意义的事实（数据落库、事件卡、信号、建议单、订单、成交、风控触发、告警、人工确认）。它是审计、回放、模型评估的唯一事实来源。
2. **指标层**：06 §2 的 P1–P9 及运行指标（延迟/成本/成功率），落 `metrics` 表，UI"系统健康"页展示，超阈自动告警。
3. **日志层**：loguru 结构化 JSON（字段含 run_id/event_id 以便与账本互查），按天轮转。日志用于排错细节，**不承担业务事实职责**——账本里没有的事等于没发生。

## 2. 来源凭证信封（Provenance Envelope）

每条事件的固定外壳，缺任一必填字段即写入失败：

```python
class Event(BaseModel):
    event_id: str            # ULID，时间有序
    ts: datetime             # 统一以 UTC 存储；展示层按北京时间换算
    kind: Literal["data_ingested","event_card","signal","proposal",
                  "order","fill","risk_trigger","alert","human_action",
                  "job_status","model_inference"]
    payload: dict            # 各 kind 的业务体，schema 见附录 A
    run_id: str              # 当日管道运行号（correlation）
    causation_id: str | None # 直接引发本事件的上游 event_id（构成因果链）
    dedup_key: str           # 幂等键（T8 依据），如 f"{kind}:{date}:{symbol}:{strategy}"
    provenance: Provenance   # ↓

class Provenance(BaseModel):
    git_sha: str             # 部署代码版本
    config_hash: str         # 生效配置的规范化哈希
    data_manifest_id: str    # 本事件所见数据世界（§3）
    prompt_version: str|None # LLM 类事件必填
    model_id: str|None       # LLM/未来 NN 类事件必填（含供应商与版本）
    seed: int|None           # 任何随机过程必填
```

因果链示例：`data_ingested → event_card → proposal → human_action(confirm) → order → fill`，UI 上任一成交可一键展开完整链路（每跳附凭证）。

## 3. 数据快照：不可变分区 + 清单（Manifest）

- 行情/财务/公告落 Parquet 分区后**永不原地修改**；上游修正历史（如复权因子追溯）时，追加"勘误分区"。
- `data_manifests(manifest_id, as_of_date, created_at, items…)`，items = (dataset, partition_path, row_count, content_hash)。每日更新任务结束生成当日 manifest；**当天所有决策事件引用同一 manifest_id**。
- 由此获得轻量双时态能力：**事件时间**（行情属于哪天）与**认知时间**（系统何时知道）分离。`yquant asof --date 2026-06-10 --symbol AAPL` 可回答"6 月 10 日决策时，系统眼中的苹果（AAPL）是什么样"——用当日 manifest 读数，而非用今天的库。
- 勘误发生时：旧 manifest 回放结果必须不变（T11），新旧差异自动生成《数据修正影响报告》（涉及的历史信号/成交是否会因新数据而不同），报告入账本。

## 4. 回放引擎与运行摘要（Run Digest）

- 每日管道收尾计算 **run digest**：当日全部事件按 event_id 排序后逐条哈希的 Merkle 根，写入 `job_runs.digest`。
- `yquant replay --run-id R --strict`：按 R 的 manifest、git_sha 对应代码（不一致则警示）、config、seed 重演全管道，重算 digest 并与账本比对。**一致 = 逐字节还原成功**；不一致输出首个分歧事件定位（kind/symbol/字段级 diff）。
- LLM 是非确定源，处理办法：推理时原始 request/response 全文落盘（`llm_raw/`，密钥脱敏，保留 180 天），回放默认**重放录音**（读取原始 response）保证确定性；`--live-llm` 才真调 API，用于评估模型漂移而非还原。
- 回放的三种日常用法：①周抽检 3 个 run 验证 P6；②发布前在 golden 数据上回放对照；③事故取证（§6）。

## 5. 告警规范

告警必须**可行动**：每条告警绑定 runbook 段落号（08 §6）。级别：S1（数据错误可能已影响决策/账务不平）→ 飞书 + 置顶横幅，S2（降级运行/指标破线）→ 飞书，S3（观察项）→ 仅 UI。同源告警 4 小时窗口去重；连续 3 日相同 S3 自动升 S2。所有告警本身也是事件（kind=alert），故告警风暴亦可回放分析。

## 6. 事故取证五步法（Incident Forensics Playbook）

任何"结果不对劲"（错误成交建议、漏报红旗、账不平、离奇净值）一律走此流程，禁止直接改数据"修平"：

1. **冻结**：标记涉事 run 为 `under_investigation`，试点期内触发新开仓暂停（08 §5 熔断联动）。
2. **取证**：一条命令导出证据包 `yquant incident collect --run-id R`——事件链、manifest、相关日志切片、LLM 原文、配置快照。
3. **重演**：`replay --strict` 确认可复现；不可复现本身即重大发现（非确定性缺陷，升 P0）。
4. **二分定位**：沿因果链回溯首个错误事件；再判层次——数据层（对照双源与旧 manifest）/ 代码层（对照上一 tag 回放）/ 配置层（config diff）/ 模型层（换 prompt 版本或录音重放对照）。
5. **归档**：`incidents` 表 + 报告模板（现象/影响面/根因/时间线/涉事事件清单/修复/防复发动作/新增测试编号）。**每个 P0/P1 事故必须沉淀 ≥1 条新测试**，编号回填报告。

## 7. 存储与保留策略

`decision_events`（永久，估算 <100MB/年）、`data_manifests`（永久）、Parquet 分区（永久，含勘误链）、`llm_raw/`（180 天）、结构化日志（90 天）、`incidents`/演练台账（永久）。全库每周冷备份一份至独立磁盘/网盘目录，备份校验哈希。

## 8. 验收（对应 WP7）

信封强制性：缺 provenance 字段的写入被拒（单元测试）；因果链 UI 演示（从一笔模拟成交回溯到原始数据分区）；`asof` 与《数据修正影响报告》在构造的勘误场景下演示；T11、T13 全绿；周抽检回放脚本就位；事故五步法在一次注入的合成缺陷上完整走通并产出合格报告。
