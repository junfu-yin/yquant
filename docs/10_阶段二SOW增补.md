## 1. 新增工作包

### WP7 · 可观测与回放地基（7–9 人日）｜规格：07 篇
账本 DDL 与写入层（信封强校验、幂等键）、manifest 机制与勘误流程、run digest、replay/asof/incident-collect 三个 CLI、告警框架、事故五步法工具化、T11/T13。
**Gate**：07 §8 全项，含合成缺陷上的取证演练报告。
**排期强约束**：**必须紧随 WP2 之后、先于 WP3–WP5 执行**。理由：信封与账本对各业务模块是侵入性架构约束，后补等于全量重构；策略/简报/纪律模块从第一行代码起就应"生在账本里"。

### WP8 · 测试体系与金标准数据（5–7 人日）｜规格：06 篇
P1–P9 指标脚本化与健康面板、属性测试（双 Broker ≥10 万随机用例）、T8–T12、golden 数据集制作与哈希入库、混沌注入脚本、CI 矩阵与发布门禁、迁移脚本机制。
**Gate**：06 §9 全项（T7 除外，归 WP9）。

### WP9 · PaperBroker 与上线阶梯流水线（7–9 人日）｜规格：08 篇
共享约束层重构（回测/模拟同源）、PaperBroker 全生命周期（撮合/估值/分红送转/对账/冻结联动）、shadow 与 paper 两种模式管道、T7 双引擎等价、孪生对照报表、执行回填与滑点闭环（为 L3 备）。
**Gate**：T7 全绿；L1 影子 20 交易日达标报告；L2 首月（可用 golden 数据加速模拟）对账零差异；拒单/降级场景演示。

### WP10 · 模型评估与可解释性框架（6–8 人日）｜规格：09 篇
Provider 协议与现有四个信号源改造、ModelCard 与解释契约（含账本联动校验）、离线评估流水线与三基线、四件套面板、OOD 与冻结阶梯、行为测试入 CI、冠军-挑战者流程工具化。
**Gate**：09 §8 全项。

### WP11 · 上线运营与演练（3–4 人日）｜规格：08 §4/§6、06 §6
运行手册全文（告警条目↔runbook 段落一一对应）、《预期区间书》模板+首份实例、消防演习×1 与混沌演练×1（另一次在陪跑期由委托人主持）、L2→L3 Gate 评审材料包模板。
**Gate**：08 §7 WP11 项；委托人按手册独立完成一次日检+一次告警处置（录屏为证）。

## 2. 与阶段一工作包的整体排期（修订版单线程顺序）

WP0 → WP1 → WP2 → **WP7** → WP3 → WP8 →（WP4 ∥ WP9，二者无依赖可并行；单人则先 WP9）→ WP5 → WP10 → WP11 → **L1 影子（≥20 交易日，期间仅修缺陷）→ L2 模拟盘（≥60 交易日）→ Gate 评审 → L3 试点**。

阶段二新增工时合计 **28–37 人日**；两阶段合计约 63–83 人日。L1/L2 为日历时间（约 4 个月），与收尾开发部分重叠，人力投入为响应制。

## 3. 验收证据的统一形态

自本增补起，一切 Gate 证据以**账本导出包**为准（事件链 + manifest + digest + 相关报告），录屏仅作辅助。委托人验收时的标准动作是：抽一条结论 → 要求 `replay --strict` 现场重演。**重演不出来的成果不存在。**

## 4. 付款里程碑建议（增补）

WP7 15% · WP8 10% · WP9 15% · WP10 15% · WP11 5%，余 40% 挂 L2 完成（模拟盘 60 交易日机器门槛全绿）后支付——将承接方利益与"稳定运行"而非"功能完工"绑定。比例为建议，双方合同为准。
