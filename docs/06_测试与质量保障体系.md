# 06 · 测试与质量保障体系

> yquant 阶段二文档 · 编号 06 · v1.0 · 2026-07-05
> 依赖：03 §5–§8（模块规格与陷阱测试 T0–T6）、07 篇（回放能力是多项测试的载体）。
> 立场：**质量不是测出来的，是被架构和门禁逼出来的。** 本篇定义"高精度"与"稳定运行"的量化含义、覆盖它们的测试手段、以及任何代码进入模拟盘/试点前必须通过的门禁。

---

## 1. 质量目标的两个词各自意味着什么

**高精度** = 系统输出的每一个数字都经得起对账：行情数据与独立来源一致、账务分毫不差、复权无跳变、LLM 卡片无编造、同一输入永远得到同一输出。
**稳定运行** = 无人值守场景下：任务按时完成、失败被降级而非扩散、崩溃后可恢复到一致状态、告警在人需要介入时且仅在需要介入时响起。

## 2. 精度指标表（ADR-15 的落地，全部纳入 CI 或日检，降级即发布阻断/告警）

| 编号 | 指标 | 目标 | 检验方式 | 频率 |
|---|---|---|---|---|
| P1 | 账务守恒：现金+持仓市值变动 = 成交额±费用，逐笔逐分 | 误差 = ¥0.00 | 属性测试（§4）+ 每日对账任务 | 每日 |
| P2 | 模拟账户 NAV 与持仓重估独立复算一致 | 误差 = ¥0.00 | 日终双算比对 | 每日 |
| P3 | 行情双源一致率：随机抽 50 股×20 日，AkShare vs 备源收盘价 | ≥99.5%（差异>0.1% 记不一致） | 周任务 | 每周 |
| P4 | 复权净值连续性（除权除息日） | 跳变 <0.1% | 陷阱 T5 + 日检新除权样本 | 每日 |
| P5 | 事件卡数字回验漏杀 | = 0 | 评测集 + 生产抽检 | 每次发布/每周 |
| P6 | 决策回放一致率（07 篇 run digest 比对） | = 100% | 陷阱 T13 + 周抽 3 日回放 | 每周 |
| P7 | 简报送达 SLA：交易日 17:30 前 | ≥98%（月度） | job_runs 统计 | 每月 |
| P8 | 管道任务成功率（含降级成功） | ≥99%（滚动 30 日） | job_runs 统计 | 每日 |
| P9 | 数据新鲜度：日线落库时间中位 | ≤16:45 | quality_reports | 每日 |

指标现值在 UI"系统健康"分区常驻展示；连续 2 日不达标自动飞书告警。

## 3. 测试金字塔与陷阱测试扩展（T7–T13）

层次：单元（pytest，覆盖率 ≥80% 于 data/backtest/discipline 核心包）→ 属性测试（§4）→ 集成（模块间契约）→ 端到端（全管道日演练）→ **陷阱测试**（给系统下毒验证其吐出）。T0–T6 已定义于 03 §8，本包新增：

| 编号 | 陷阱 | 方法 | 通过判据 |
|---|---|---|---|
| T7 | 双引擎撮合等价 | 同一策略同一区间：backtest 引擎一次性运行 vs PaperBroker 逐日推进（08 §3），两条净值曲线比对 | 逐日差 ≤2bps、区间累计差 ≤20bps；超差输出首个分歧日定位 |
| T8 | 幂等性 | 同一交易日管道连跑 3 次 | 事件账本无重复（同 dedup key 仅一条）、订单/卡片/推送不重复 |
| T9 | 崩溃恢复 | 管道各阶段（拉数中/撮合中/写库中）注入 kill -9，重启后重跑 | 状态一致（P1/P2 达标），无半写脏数据（事务/临时文件+原子改名） |
| T10 | 时钟与时区 | 伪造系统时区为 UTC/US Eastern、伪造 NTP 漂移 ±5min | 全部业务时间仍按 Asia/Shanghai 正确；漂移超阈值时任务拒跑并告警 |
| T11 | 上游数据修正 | 模拟数据源修改历史价格（复权因子追溯调整场景） | 旧 manifest 回放结果不变；新旧 manifest 差异被自动报告而非静默覆盖 |
| T12 | 熔断有效性 | 注入触发回撤熔断线的行情 | 新开仓 proposal 被拒并记 risk_events；解除需人工操作留痕 |
| T13 | 回放确定性 | 抽任意历史 run，`replay --strict` | run digest 逐字节一致（07 §4） |

## 4. 属性测试（hypothesis）：账务与撮合的不变量

对随机生成的订单流/费率/价格序列断言以下不变量恒成立：①资金守恒（P1 公式）；②持仓非负、可卖数量 ≤ 持仓且遵守 T+1；③费用非负且 ≥ 最低佣金规则；④任何成交价 ∈ 当日 [low, high] 且不违反涨跌停约束；⑤先进先出的成本核算与浮盈计算自洽。属性测试是"高精度"最廉价的守门员，要求覆盖 PaperBroker 与 ChinaBroker 两套实现。

## 5. 金标准数据集（Golden Dataset）

冻结一份 2019-01-01–2021-12-31、约 500 只（含 ≥30 只期内退市、≥20 个除权除息事件、≥3 个一字涨停样本日、≥2 段长停牌）的数据切片，内容哈希入库（manifest 形式，07 篇）。用途：①陷阱与回归测试的固定基准；②新数据源适配器的验收对照；③评测集（03 §5.4、09 §3）的行情侧依据。golden 数据只可追加勘误分区、不可原地修改，勘误需 issue 留痕。

## 6. 混沌演练与消防演习（稳定运行的实战检验）

- **混沌注入（每月一次，脚本化）**：随机选一项——断网 10 分钟于拉数窗口 / 主数据源返回 500 / LLM 超时率 50% / 磁盘写满模拟 / SQLite 锁竞争。判据：系统按 03 §5.1 降级链路运行、告警一次且仅一次、次日自动补齐、事件账本完整记录该事故。
- **消防演习（每月一次，含人）**：注入假场景——伪造某自选股 severity=5 红旗公告 / 伪造持仓一字跌停 / 伪造熔断触发。判据：推送在 15 分钟内到达、UI 红旗正确展示、人按 08 §6 运行手册完成既定动作、全程可在事后回放复盘。演习台账入库，作为 L2→L3 Gate 材料（08 §5）。

## 7. LLM 专项质量

评测集从 60 条扩至 ≥120 条（新增：数字密集型公告 30、易混淆事件对 20、诱导幻觉样本 10——如原文无具体金额但标题暗示金额）；每次 prompt/模型变更跑全量回归，任一指标较基线降 >2pp 即回滚；生产侧每周随机抽 20 张卡片人工复核并记录抽检准确率趋势；模型供应商切换视同发布，走同样门禁。

## 8. CI/CD 与发布门禁

- CI 矩阵：lint(ruff)+类型(mypy)+单元+属性 → 陷阱 T0–T13 → LLM 评测回归（有 prompt/模型变更时）→ golden 端到端（全管道在 golden 数据上跑一个模拟日）。
- 发布 = 打 tag + CHANGELOG + 数据库迁移脚本（`migrations/` 顺序号 SQL，禁止手改生产库）+ 回滚说明。模拟盘与试点环境仅部署 tag，不部署 main 头。
- **门禁红线**：跳过或注释任何 T 系列测试的提交，验收时按 P0 缺陷处理（延续 03 §11 技术债红线）。

## 9. 验收（对应 WP8）

06 篇全部指标脚本化并在 CI/日检中运行；T7–T13 全绿演示；属性测试对两套 Broker 通过 ≥10 万随机用例；golden 数据集交付（含清单文档与哈希）；一次完整混沌演练+一次消防演习的台账。
